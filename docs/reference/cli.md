# CLI Reference

Complete reference for all RiverGen commands and flags.

For concepts referenced here, see [docs/concepts/](../concepts/).

---

## Commands

| Command                    | Argument                  | Purpose                                                     |
| -------------------------- | ------------------------- | ----------------------------------------------------------- |
| `rivergen init`            | none                      | Write infrastructure framework files (run once per project) |
| `rivergen plan <specFile>` | spec file path (required) | Dry-run: print plan without writing                         |
| `rivergen gen <specFile>`  | spec file path (required) | Write domain files and regenerate barrels                   |
| `rivergen verify`          | none                      | Run all 12 gates against the project                        |

**Validation rules:**

- `gen` and `plan` require a `<specFile>` positional argument. Omitting it is a fatal error.
- `init` and `verify` do not accept a `<specFile>` argument.
- Passing any command other than `init`, `plan`, `gen`, or `verify` prints usage and exits with code 1.

---

## rivergen init

Writes the static infrastructure that every RiverGen project requires. Run once per project, before the first `rivergen gen`.

```sh
rivergen init
rivergen init --plan          # dry-run: show what would be written, then exit
rivergen init --force         # overwrite all existing files
```

### Files written (17 total)

**Static infrastructure** — written once by `init`, never modified by `gen`:

| File                                                      | Description                                      |
| --------------------------------------------------------- | ------------------------------------------------ |
| `apps/api/src/lib/event-factory/event-factory.service.ts` | EventFactory service                             |
| `apps/api/src/lib/event-bus.service.ts`                   | EventBus service                                 |
| `apps/api/src/lib/event-factory/schemas/_base.ts`         | Base Zod schemas for event envelopes             |
| `apps/web/src/lib/cache/entity-cache.ts`                  | `applyEntityCreate/Update/Delete` helpers        |
| `apps/web/src/lib/cache/state-cache.ts`                   | `applyRealtimeEventToCache` dispatch entry point |
| `apps/web/src/providers/WebSocketProvider.tsx`            | Socket.io connection and event routing           |
| `packages/shared/src/entity-projections/_types.ts`        | `EntityProjectionEntry` type definitions         |

**Barrel stubs** — written by `init`, regenerated by every `gen` run:

| File                                                  |
| ----------------------------------------------------- |
| `apps/api/src/lib/event-factory/schemas/_index.ts`    |
| `apps/web/src/lib/cache/domain-dispatchers/_index.ts` |
| `apps/web/src/providers/ws-bindings/_index.ts`        |
| `apps/web/src/lib/query-keys/_index.ts`               |
| `packages/shared/src/entity-projections/_index.ts`    |

**Agent rules files** — written to project root:

| File        | Description                                     |
| ----------- | ----------------------------------------------- |
| `AGENTS.md` | One River rules for AI coding agents            |
| `CLAUDE.md` | Auto-loads `AGENTS.md` into Claude Code context |

**Vite proxy shims** — thin `export * from "./_index"` files that bridge directory imports to `_index.ts` barrels:

| File                                                 |
| ---------------------------------------------------- |
| `apps/web/src/lib/query-keys/index.ts`               |
| `apps/web/src/lib/cache/domain-dispatchers/index.ts` |
| `apps/web/src/providers/ws-bindings/index.ts`        |

### Idempotency

`rivergen init` aborts if any of the 17 files already exist. Use `--force` to overwrite.

The `--plan` flag shows the full file manifest and marks any existing files, then exits without writing anything.

## rivergen plan \<specFile\>

Validates the spec file, checks dependencies, and prints what `rivergen gen` would do. Does not write any files.

```sh
rivergen plan specs/task.json
rivergen plan specs/task.json --root /path/to/project
```

### Output sections

| Section                     | Content                                                                                            |
| --------------------------- | -------------------------------------------------------------------------------------------------- |
| `FILES TO CREATE`           | 12 domain files. Files that already exist are marked `← EXISTS (blocked until --force)`.           |
| `BARRELS TO REGENERATE`     | The 5 barrel `_index.ts` files that `gen` always rewrites.                                         |
| `SATELLITE FILES TO UPDATE` | Optional project-specific files detected in the repo (shown only when present).                    |
| `DEPENDENCIES`              | Required packages. Missing packages are listed with a prompt to use `--install`.                   |
| `READY / NOT READY`         | Summary. Not ready if the spec is invalid, files exist without `--force`, or packages are missing. |

### Exit codes

- `0` — plan printed (even if NOT READY due to existing files or missing deps)
- `1` — spec file not found or JSON parse failure (no plan can be produced)

### What to check before running gen

1. All 12 FILES TO CREATE show `✓` (none marked `← EXISTS`).
2. DEPENDENCIES shows `✓  All required packages present`, or you will run `gen --install`.
3. No spec validation errors in the NOT READY section.

## rivergen gen \<specFile\>

Generates all domain files for one spec and regenerates the 5 barrel `_index.ts` files.

```sh
rivergen gen specs/task.json
rivergen gen specs/task.json --force
rivergen gen specs/task.json --install
rivergen gen specs/task.json --force --install --root /path/to/project
```

`gen` prints the full plan output before writing, so you see the same information as `rivergen plan` before any file is touched.

### Domain files written (12)

One set per domain. `<domain>` is the kebab-case domain key from the spec (e.g., `task`, `work-order`).

These files are **not overwritten without `--force`**:

| File                                                        | Fill required                                                                   |
| ----------------------------------------------------------- | ------------------------------------------------------------------------------- |
| `apps/api/src/<domain>/<domain>.router.ts`                  | API endpoint URLs                                                               |
| `apps/api/src/<domain>/<domain>.mutations.ts`               | DB calls, input validation, payload construction                                |
| `apps/api/src/<domain>/<domain>.broadcast.ts`               | Nothing when `privateRoomTemplate` is set; private room expression if omitted   |
| `apps/api/src/lib/event-bus-listeners/<domain>.listener.ts` | Pure wiring — no fill required                                                  |
| `apps/api/src/lib/event-factory/schemas/<domain>.ts`        | Actual payload fields beyond the `entityId` stub                                |
| `apps/web/src/lib/projections/<domain>-projections.ts`      | `context` object matching entity-projection slice                               |
| `apps/web/src/hooks/use-<domain>.ts`                        | API call URLs, entity types                                                     |
| `apps/web/src/lib/cache/domain-dispatchers/<domain>.ts`     | Pure wiring — no fill required                                                  |
| `apps/web/src/providers/ws-bindings/<domain>.ts`            | Pure wiring — no fill required                                                  |
| `packages/shared/src/entity-projections/<domain>.ts`        | Rarely — room-var-aware key factory is generated correctly from room template   |
| `apps/web/src/lib/query-keys/<domain>.ts`                   | Rarely — generated correctly from room template                                 |
| `apps/web/src/witness/<domain>.witness.ts`                  | Payload interface, `requiredFields`, `testPayloads`, `lifecycle()`, `signals{}` |

For fill order guidance, see [docs/guides/fill-order.md](../guides/fill-order.md).

### Barrel files regenerated (5)

These are **always overwritten** on every `gen` run. It is safe to re-run `gen` at any time without `--force` — only these files will be touched if domain files already exist:

- `apps/api/src/lib/event-factory/schemas/_index.ts`
- `apps/web/src/lib/cache/domain-dispatchers/_index.ts`
- `apps/web/src/providers/ws-bindings/_index.ts`
- `apps/web/src/lib/query-keys/_index.ts`
- `packages/shared/src/entity-projections/_index.ts`

### --force

Overwrites all 12 domain files for the spec, in addition to regenerating the 5 barrels. Use when re-scaffolding a domain after changing the spec.

### --install

Before writing files, installs any missing required packages into the appropriate workspaces via `pnpm`. Without this flag, missing packages are reported as warnings and generation proceeds; the generated code will have unresolved imports until the packages are installed.

## rivergen verify

Runs all 12 gates against the project at the current working directory (or `--root`).

```sh
rivergen verify
```

There are no positional arguments. `verify` does not accept a spec file.

### Gate inventory

| #   | Gate                                                                 | Type                           | Skip-eligible                                               |
| --- | -------------------------------------------------------------------- | ------------------------------ | ----------------------------------------------------------- |
| 1   | Mutation → EventFactory (no socket.emit / eventBus bypass)           | Synchronous                    | No                                                          |
| 2   | Event → Listener → Broadcaster → socket.emit                         | Synchronous                    | No                                                          |
| 3   | WS socket.on → Dispatcher → Projection call                          | Synchronous                    | No                                                          |
| 4   | Projection → entity-cache helpers                                    | Synchronous                    | No                                                          |
| 5   | Broadcast room scoping (PRIVATE → scoped rooms)                      | Synchronous                    | No                                                          |
| 6   | EventFactory schema coverage (every emitted event has a schema)      | Synchronous                    | No                                                          |
| 7   | Schema `.strict()` enforcement                                       | Synchronous                    | No                                                          |
| 8   | Provider isolation (no entity-cache in WebSocketProvider)            | Synchronous                    | No                                                          |
| 9   | No cache writes in `onSuccess`                                       | Synchronous                    | No                                                          |
| 10  | Optimistic coverage (every `useMutation` has `onMutate` + `onError`) | Synchronous                    | No                                                          |
| 11  | Audit coverage (Phase 4/5/6 audit files)                             | Synchronous                    | Yes — skips silently if none of the three audit files exist |
| 12  | Witness coverage (Layer 1 + 2 + 3 + 4)                               | Async (subprocess for Layer 3) | Yes — skips if `witnessDir` does not exist                  |

Gates #1–#11 are synchronous static analysis. Gate #12 is asynchronous: Layer 3 runs projection functions in a subprocess to isolate React-dependent imports.

For a detailed description of what each gate checks and how to fix violations, see [docs/reference/gates.md](gates.md).

For Witness layers, see [docs/concepts/witness-layers.md](../concepts/witness-layers.md).

### Output format

```
  RiverGen — Gate Verification Report
  Project: /path/to/project
  Run at:  <ISO timestamp>

────────────────────────────────────────────────────────────────────────
  ✓  Gate #1 — Mutation → EventFactory
     <summary line>

────────────────────────────────────────────────────────────────────────
  ✗  Gate #9 — No cache writes in onSuccess
     <summary line>

     ✗  [ERROR] apps/web/src/hooks/use-task.ts:42
            queryClient.invalidateQueries() inside onSuccess is forbidden.
```

Each gate entry shows:

- Pass (`✓`), fail (`✗`), warning-only pass (`⚠`), or skipped (`○`)
- Gate name and summary line
- Per-violation entries: file path, optional line number, severity (`[ERROR]` or `[WARN]`), message
- Informational notes (e.g., Gate #12 Layer 3 stub status) appear as `⚠` lines below the summary

### Exit codes

| Code | Condition                                                                                       |
| ---- | ----------------------------------------------------------------------------------------------- |
| `0`  | All gates passed. Skipped gates count as passing. Warning-only violations do not cause failure. |
| `1`  | One or more gates have at least one `error`-severity violation.                                 |

## Flag reference

| Flag           | Type    | Commands      | Default         | Description                                                                                                                                                        |
| -------------- | ------- | ------------- | --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `--force`      | boolean | `init`, `gen` | `false`         | Overwrite existing files. On `init`: overwrites all 17 infrastructure files. On `gen`: overwrites all 12 domain files (barrels are always overwritten regardless). |
| `--install`    | boolean | `gen`         | `false`         | Auto-install missing required packages via `pnpm` before generating files.                                                                                         |
| `--root <dir>` | string  | all commands  | `process.cwd()` | Project root directory. All file paths are resolved relative to this directory.                                                                                    |
| `--plan`       | boolean | `init`        | `false`         | Dry-run for `init`: prints the file manifest showing what would be written, then exits with code 0 without writing anything.                                       |

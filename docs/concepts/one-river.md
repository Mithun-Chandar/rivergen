# One River

## The invariant

RiverGen enforces a single rule above all others: **there is exactly one verified path from a mutation to the TanStack Query cache.**

That path is:

```
mutation
  → EventFactory.publish()
    → EventBus (Node.js EventEmitter)
      → EventBus listener (subscribe)
        → broadcast helper (io.to(room).emit)
          → WebSocketProvider (socket.on via getAllWsBindings loop)
            → applyRealtimeEventToCache (state-cache dispatcher)
              → domain dispatcher (domainDispatchers[eventName])
                → projection function (apply*Created/Updated/Deleted)
                  → entity-cache helpers (applyEntityCreate/Update/Delete)
                    → TanStack Query cache ✓
```

Every component in that path is generated by `rivergen gen`. Every connection between them is enforced by a gate.

---

## The path in detail

### 1. Mutation → EventFactory

The mutation function calls `eventFactory.publish()` with a typed input:

```typescript
await eventFactory.publish({
  type: "task.created",
  resourceId: task.id,
  actor: { id: userId, type: "user" },
  context: { realmId: projectId },
  correlationId: randomUUID(),
  eventVersion: "1.0",
  payload: { taskId: task.id, title: task.title, projectId },
});
```

**Gate #1** enforces this: every `*.mutations.ts` file must import and call `eventFactory.publish()`. Direct `socket.emit()`, `io.to().emit()`, or `eventBus.publish()` calls in a mutation file are gate errors.

### 2. EventFactory → EventBus

EventFactory validates the payload against the domain's Zod schema (`EventPayloadSchemas["task.created"]`), wraps it in an `EventEnvelope`, validates the full envelope, then calls `eventBus.publishEnvelope(envelope)`.

The EventBus extends Node.js `EventEmitter`. `publishEnvelope` emits on both `envelope.type` (e.g. `"task.created"`) and the wildcard `"*"` — listeners subscribe to either.

**Gate #6: EventFactory Schema Coverage** and **Gate #7: Schema .strict() Enforcement** enforce the schema side: every emitted event must have a registered `.strict()` Zod schema. Without `.strict()`, the payload schema accepts unknown fields, bypassing the guarantee that only declared fields reach the bus.

### 3. EventBus → Listener → Broadcast

The domain listener subscribes to each event and passes the payload to the broadcast helper:

```typescript
eventBus.subscribe("task.created", (envelope) => {
  broadcastTaskEvent(io, "task.created", envelope.payload as AnyPayload);
});
```

The broadcast helper resolves the socket.io room from the payload and emits:

```typescript
io.to(room).emit(eventName, payload);
```

Note: only `envelope.payload` travels over the socket, not the full envelope. The `actor`, `context`, `correlationId`, and other envelope fields stay on the server side.

**Gate #2** enforces this: every event registered in `EventPayloadSchemas` must have a listener (`eventBus.subscribe`) and a broadcast emit. **Gate #5: Broadcast Room Scoping (PRIVATE entities → scoped rooms)** enforces room scoping: bare `io.emit()` with no room is forbidden, and private entities must not be emitted to workspace-wide rooms without a visibility guard.

### 4. WebSocket → Provider → Dispatcher

`WebSocketProvider` (written once by `rivergen init`) uses `getAllWsBindings()` to read all domain event names from the `ws-bindings/` slice files and registers a handler for each:

```typescript
for (const event of getAllWsBindings()) {
  socket.on(event, routeEvent(event));
}
```

`routeEvent(eventName)(payload)` calls:

```typescript
applyRealtimeEventToCache(eventName, payload, queryClient);
```

`applyRealtimeEventToCache` (in `state-cache.ts`, written once by `rivergen init`) does a lookup:

```typescript
const handler = domainDispatchers[eventName];
if (handler) {
  handler(payload, queryClient);
} else if (process.env.NODE_ENV !== "production") {
  console.warn("[Dispatcher] No handler registered for event:", eventName);
}
```

`domainDispatchers` is the merged flat map from all `domain-dispatchers/<domain>.ts` slice files, regenerated on every `rivergen gen`.

**Gate #3** enforces this: every WS event bound in the provider must appear in a domain dispatcher, and the dispatcher entry must call an `apply*Projection()` function. **Gate #8: WebSocketProvider Entity-Cache Isolation** enforces provider isolation: `WebSocketProvider` must not import `entity-cache` or call `applyEntity*()` directly — all domain events must route through `applyRealtimeEventToCache`.

### 5. Dispatcher → Projection → entity-cache

Each domain dispatcher slice maps event names to projection calls:

```typescript
export const taskDispatchers = {
  "task.created": (payload, qc) => applyTaskCreated(payload, qc),
  "task.updated": (payload, qc) => applyTaskUpdated(payload, qc),
  "task.deleted": (payload, qc) => applyTaskDeleted(payload, qc),
};
```

The projection function calls an entity-cache helper:

```typescript
export function applyTaskCreated(
  payload: AnyPayload,
  queryClient: QueryClient,
): void {
  const taskId = payload.taskId as string | undefined;
  if (!taskId) return;
  const context = {
    projectId: payload.projectId as string,
    clientTempId: payload.clientTempId as string | undefined,
  };
  applyEntityCreate("task", { id: taskId, ...payload }, context, queryClient);
}
```

`applyEntityCreate` uses `setQueriesData` with a key-prefix predicate to update all matching cache entries without requiring knowledge of the exact query key structure.

**Gate #4** enforces this: every `*-projections.ts` file must import and call the entity-cache helpers. Raw `queryClient.setQueryData()` calls in projection files are gate errors (exception: files annotated with `// gate4:map-projection`).

### 6. Cache → UI

The TanStack Query cache update triggers a re-render in any component subscribed to the affected query keys. If the mutation used `onMutate` to insert an optimistic ghost, `applyEntityCreate` removes the ghost first (via `clientTempId` matching) before inserting the real entity.

**Gate #9: No Cache Writes in onSuccess** enforces no `onSuccess` cache writes: server truth arrives via the WebSocket projection; writing to cache in `onSuccess` creates a dual-write race. **Gate #10: Optimistic UI Coverage (onMutate + onError)** enforces optimistic coverage: every domain `useMutation` must have both `onMutate` and `onError`.

## Why two paths break things

When there is exactly one path, every cache update for a given entity arrives in a single place — the projection. If data is wrong at the cache, the investigation is deterministic: check the schema, check the broadcast, check the projection.

When there are two paths — an `onSuccess` write and a WS projection — both run after a mutation. Their execution order depends on network timing. The `onSuccess` fires when the HTTP response arrives. The WS event fires when the server emits after processing. On a fast connection they are nearly simultaneous. On a slow connection they can arrive seconds apart in either order. The last writer wins, and "last" is non-deterministic.

Worse: when something goes wrong, the investigation must consider both paths. Did the `onSuccess` write the right data? Did the projection overwrite it? Did the `onSuccess` run before or after the WS event? Teams that have lived through this end up wrapping the entire realtime layer in fear.

One River eliminates the question. The projection owns the cache. The `onSuccess` is intentionally omitted. There is nothing to race against.

## What the gates protect

| Gate     | Which step it protects                                         |
| -------- | -------------------------------------------------------------- |
| Gate #1  | mutation → `EventFactory.publish`                              |
| Gate #2  | event → listener → broadcaster → `socket.emit`                |
| Gate #3  | `socket.on` → dispatcher → projection call                    |
| Gate #4  | projection → entity-cache helpers                              |
| Gate #5  | broadcast room scoping (PRIVATE entities → scoped rooms)      |
| Gate #6  | EventFactory schema coverage                                   |
| Gate #7  | schema `.strict()` enforcement                                 |
| Gate #8  | WebSocketProvider entity-cache isolation                       |
| Gate #9  | no cache writes in `onSuccess`                                 |
| Gate #10 | optimistic UI coverage (`onMutate` + `onError`)                |
| Gate #11 | event audit coverage                                           |
| Gate #12 | Witness — field continuity coverage                            |

See [docs/reference/gates.md](../reference/gates.md) for the full gate reference.

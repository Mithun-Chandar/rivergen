# Event Envelope

## What EventFactory produces

Every call to `eventFactory.publish()` results in an `EventEnvelope` â€” a validated wrapper that carries both the domain-specific payload and the metadata needed to trace the event through the system.

The envelope shape (from `schemas/_base.ts`):

```typescript
{
  id: string;            // UUID generated by randomUUID()
  type: string;          // event name: "task.created"
  envelopeVersion: string; // always "1.0"
  eventVersion: string;  // from publish input â€” your schema version
  timestamp: number;     // Date.now() in milliseconds
  resourceId: string;    // the primary entity's ID
  actor: {
    id: string;
    type: "user" | "system" | "service";
    name?: string;
    email?: string;
  };
  context: {
    realmId: string;     // always required â€” the workspace/realm scope
    projectId?: string;
    channelId?: string;
    userId?: string;
  };
  payload: unknown;      // the domain-specific data, validated by domain schema
  correlationId: string; // from publish input â€” for tracing across services
  origin: {
    service: string;     // process.env.SERVICE_NAME ?? "api"
    instanceId: string;  // process.env.HOSTNAME ?? "local-dev"
  };
}
```

All fields are validated with Zod `.strict()` at both the payload level (domain schema) and the envelope level (`EventEnvelopeSchema`).

## What travels over the socket

The full `EventEnvelope` is **not** sent to the frontend. The broadcast helper extracts only `envelope.payload` and emits that:

```typescript
// In the listener:
eventBus.subscribe("task.created", (envelope) => {
  broadcastTaskEvent(io, "task.created", envelope.payload as AnyPayload);
});

// In the broadcast helper:
io.to(room).emit(eventName, payload); // payload only â€” not the envelope
```

The frontend receives only the domain payload object (e.g. `{ taskId, title, projectId, clientTempId }`). The `actor`, `context`, `correlationId`, `origin`, and other envelope fields stay on the server.

This matters for Witness: `requiredFields` and `testPayloads` describe what the frontend actually receives â€” the payload object, not the full envelope.

## What EventFactory does step by step

```typescript
async publish<TPayload>(input: PublishInput<TPayload>): Promise<void> {
  // 1. Look up the domain's Zod schema
  const payloadSchema = EventPayloadSchemas[input.type];
  if (!payloadSchema) {
    throw new Error(`No payload schema registered for event type "${input.type}"`);
  }

  // 2. Validate and strip the payload
  const parsedPayload = payloadSchema.parse(input.payload);

  // 3. Build the envelope
  const envelope: EventEnvelope = {
    id: randomUUID(),
    type: input.type,
    envelopeVersion: "1.0",
    eventVersion: input.eventVersion,
    timestamp: Date.now(),
    resourceId: input.resourceId,
    actor: input.actor,
    context: input.context,
    payload: parsedPayload,
    correlationId: input.correlationId,
    origin: buildOrigin(),
  };

  // 4. Validate the full envelope
  EventEnvelopeSchema.parse(envelope);

  // 5. Publish to the EventBus
  await eventBus.publishEnvelope(envelope);
}
```

Step 2 is where `.strict()` matters. If a field is in `publish()` but absent from the schema, `payloadSchema.parse()` strips it silently. The stripped payload is what reaches the EventBus â€” and therefore the frontend. This is why Witness Layer 1 checks that `requiredFields` are declared in the schema.

Step 1 is why `eventBus.publish()` is banned in mutations (Gate #1). Calling the bus directly skips step 1 and 2 entirely â€” unregistered events and unvalidated payloads reach the broadcast layer.

## The EventBus

`EventBusService` extends Node.js `EventEmitter`. It is an in-process pub/sub system â€” not a distributed message queue.

```typescript
async publishEnvelope(envelope: EventEnvelopeRecord): Promise<void> {
  this.publishedEnvelopes.push(envelope); // kept in memory for test inspection
  this.emit(envelope.type, envelope);     // fires "task.created" listeners
  this.emit("*", envelope);              // fires wildcard listeners
}

subscribe(event: string | "*", handler: (envelope) => void): void {
  this.on(event, (payload) => { void handler(payload); });
}
```

Listeners receive the full `EventEnvelope`. The listener extracts `envelope.payload` before passing it to the broadcast helper.

`getPublishedEnvelopes()` and `clear()` are available for tests â€” they give direct inspection of everything published during a test run.

## The `_meta` pattern in witness test payloads

The generated `*.witness.ts` scaffold includes `_meta` inside each `testPayload` entry:

```typescript
testPayloads: {
  "task.created": {
    taskId: "test-task-001",
    // TODO: add all fields from TaskPayload
    _meta: {
      resourceId: "test-task-001",
      actor: { id: "user-test-01", type: "user" },
      context: { realmId: "realm-test-01" },
      correlationId: "corr-task-created-01",
      eventVersion: "1.0",
    },
  },
},
```

`_meta` is a scaffold convention â€” it mirrors the envelope-level fields so test payloads are realistic. It is **not** a field that travels through the socket, and it is **not** in `requiredFields` unless your projection explicitly reads `payload._meta`. Remove or keep it based on what your projection actually reads.

The fields that matter for Witness are only the ones your projection function reads from the payload. Those go in `requiredFields`, and those are what Layer 1, Layer 2, and Layer 3 verify.

## Schema slices and the barrel

Domain payload schemas live in `apps/api/src/lib/event-factory/schemas/<domain>.ts` as named exports. The barrel `schemas/_index.ts` merges them into `EventPayloadSchemas`:

```typescript
export const EventPayloadSchemas: Record<string, z.ZodTypeAny> = {
  ...taskSchemas,
  ...invoiceSchemas,
  // ...one entry per domain
};
```

The barrel is regenerated on every `rivergen gen` call. EventFactory uses `EventPayloadSchemas` at publish time to look up the schema for any event type.

A domain schema slice looks like:

```typescript
export const taskSchemas: Record<string, z.ZodTypeAny> = {
  "task.created":  z.object({ taskId: z.string().min(1) }).strict(),
  "task.updated":  z.object({ taskId: z.string().min(1) }).strict(),
  "task.deleted":  z.object({ taskId: z.string().min(1) }).strict(),
};
```

The generated entry only includes the entity ID. You fill in additional fields as your mutation's payload grows. Every field you add to the schema must also be added to `eventFactory.publish()` in the mutation, and must match the REST API response shape (see [docs/concepts/field-shape-law.md](field-shape-law.md)).
